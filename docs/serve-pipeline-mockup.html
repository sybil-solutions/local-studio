Serve Pipeline

Replace the fractured engine / download / recipe / docker / environments / discover surface with one noun and one flow. A user can go from “I want that model” to “it is serving on my controller” without leaving Models.

Product promise: Every model you can run is a Serve — weights on disk, a chosen runtime (managed venv · system binary · Docker image), and serve settings. Launch always uses that Serve. Nothing else is a second path.

single surface runtime is real docker is a runtime kind old paths deleted
01
Get
Browse, recommend, or paste HF id. Download weights into models_dir.
02
Runtime
Pick how it runs: managed engine, system install, or Docker image. Install if missing.
03
Configure
Backend defaults + resources. Path, port, name auto-filled from download.
04
Serve
One launch path. Health ready → Status + Chat. Stop/evict same surface.

In scope (v1)

  • Models page owns Get + Serve list + editor
  • Setup wizard is a thin first-run skin over the same pipeline
  • Runtime selection is persisted and used at launch
  • Docker only as runtime kind for engines that support it (vLLM first; SGLang when image contract known)
  • Post-download always offers Create Serve

Explicitly out of scope

  • No Environments page / CRUD / start-stop dual lifecycle
  • No separate Discover page
  • No docker RuntimeTarget that cannot launch
  • No secret-only extra_args.docker_image without UI
  • No fire-and-forget second launch client
Lock rule. If a code path can put weights on disk or start an inference process without going through a Serve entity + the single launch coordinator, it is deleted.
Contracts

One entity: Serve

Rename in product language to “Serve” (UI). Wire type can stay Recipe during migration, then rename. Runtime is a first-class field — not a side channel.

Serve (canonical)

// shared/contracts/serve.ts type RuntimeKind = "managed_venv" | "system" | "docker" | "binary" interface ServeRuntime { kind: RuntimeKind // managed_venv → backend id resolves path // system/binary → absolute path // docker → image ref, e.g. vllm/vllm-openai:v0.8.5 ref: string label?: string } interface Serve { id: string name: string backend: "vllm" | "sglang" | "llamacpp" | "mlx" model_path: string runtime: ServeRuntime served_model_name?: string // resources + engine args (existing fields) max_model_len: number gpu_memory_utilization: number tensor_parallel_size: number extra_args: Record<string, unknown> env_vars?: Record<string, string> }

Runtime catalog (discovery only)

Discovered targets feed the picker. They are not launchable by themselves. Launch only happens from a Serve whose runtime was set.

interface RuntimeOption { id: string backend: EngineBackend kind: RuntimeKind label: string ref: string installed: boolean canInstall: boolean version?: string | null }

Invariants (non-negotiable)

I1
buildCommand(serve) reads serve.runtime only — never global “selected target” alone.
I2
Docker Serves set runtime.kind = "docker". No parallel extra_args.docker_image.
I3
Completed download → always can create Serve with model_path = target_dir (or GGUF file).
I4
One launch entry: POST /launch/:id → coordinator → process manager.
I5
Managed install writes a RuntimeOption the next Serve picker can select.
